Slide 1 / 封面

OpenSpec 不是多寫文件，而是把專案講清楚

這份簡報會一路往右走，快速看懂它能帶來什麼、要付出什麼。

證據來源：本倉庫 OpenSpec 結構與提示流程

傳統做法常出現

  • 需求在會議後持續漂移
  • 執行中看不到決策脈絡
  • 交付後難以回溯責任與依據

真正要解的，不只速度

  • 先解決對齊與治理
  • 再放大交付效率
  • 避免「快但不可控」
Slide 2 / 痛點

專案會失速，常常不是因為寫不出來

而是需求、決策、責任沒被排好。流程不清楚，溝通就會走樣。

證據來源：變更提案 Why 與 What Changes
01
Explore
先擴展理解，暫不實作
02
Propose
一次產生 proposal / design / tasks
03
Apply
依任務逐步落地並追蹤進度
04
Archive
封存變更並同步可用規格
Slide 3 / 全景

OpenSpec 用四步，把混亂變成有節奏的推進

不是多加流程，而是每一步都看得到、接得上、交得清楚。

證據來源：opsx-explore / propose / apply / archive prompts

Explore 的核心規則

  • 可探索問題空間、架構與風險
  • 可讀檔、可比較方案、可畫圖澄清
  • 不可直接實作功能（先想清楚）
Slide 4 / Explore

先想清楚，再動手，才不會白忙一場

先確認值不值得做、做到哪裡，後面才不會一直回頭改。

證據來源：opsx-explore.prompt.md guardrails

proposal.md

講清楚為什麼現在要做、影響到哪些地方。

design.md

說明要怎麼做，以及為什麼這樣選。

tasks.md

把工作拆成一格一格，做完就能打勾。

Slide 5 / Propose

提案階段一次把「為什麼、怎麼做、怎麼執行」說完

開工前先對齊，後面就少很多來回與重工。

證據來源：opsx-propose.prompt.md artifact 定義

Apply 任務循環

讀 context
proposal/specs/design/tasks
執行任務
一次聚焦一項變更
勾選完成
- [ ] 轉 - [x]
再檢查狀態
持續到 all done
Slide 6 / Apply

執行時一件一件完成，比一次大包交付更穩

每一步都能確認，發現偏掉也能立刻拉回來。

證據來源：opsx-apply.prompt.md state 與 loop 規則
changes/<name>
→
archive/YYYY-MM-DD-<name>
→
specs/ 主規格同步
Slide 7 / Archive

封存不只是收尾，還是把經驗留給下一次

這次做完的學習會回到主規格，下一個專案能直接少踩坑。

證據來源：opsx-archive.prompt.md delta specs sync
$ openspec new change "openspec-presentation-pros-cons-zh"
$ openspec status --change "openspec-presentation-pros-cons-zh" --json
$ openspec instructions proposal --change "openspec-presentation-pros-cons-zh" --json
$ openspec instructions apply --change "openspec-presentation-pros-cons-zh" --json
$ openspec status --change "openspec-presentation-pros-cons-zh"
示意頁：以上為流程示意（非 live demo），目的在說明管理節點與可追蹤狀態。
Slide 8 / CLI 示意

流程能用指令跑，代表它可以被標準化

主管不用盯細節，只要看狀態就知道專案目前在哪一段。

證據來源：本次實際變更流程與 openspec CLI 輸出

問題

工程受眾與主管受眾共用淺色語言，工程模式辨識度不足。

做法

建立 mode 對應 theme 的單一映射：supervisor=light，dev=dark。

命令路徑

new change → instructions apply → status all_done

結果

切換模式時，文案與主題同時切換，且不重置當前頁。

收斂

同一骨架服務雙受眾，展示一致且維運成本更低。

Slide 9 / 實作範例

用 add-dev-dark-theme，示範同骨架雙受眾如何落地

這不是換皮，而是把受眾語言與視覺情境綁在一起，讓溝通與決策更直覺。

證據來源：add-dev-dark-theme proposal / design / spec / tasks

管理更有抓手

每個階段都有明確輸入輸出，不再靠感覺追進度。

決策可回頭查

從提案到封存都有紀錄，事後能快速釐清原因。

跨角色更同頻

先對齊再開工，工程、PM、主管比較不會各說各話。

Slide 9 / 優勢

OpenSpec 最大價值，不是更快，而是更穩

團隊越大、事情越複雜，「可控」通常比「局部快」更重要。

證據來源：proposal/design 中的 goals 與 impact

CLI 相依

導入初期要花時間熟悉流程與工具。

緩解：先從小範圍試點

同步複雜

封存和規格同步要有命名與審核規則。

緩解：固定同步檢查清單

人工勾選

如果沒即時勾選，進度看板就會失真。

緩解：完成即勾選 + review gate
Slide 10 / 風險與代價

好方案要同時講清楚好處和代價

先把難點說明白，再配好對策，導入才會真的落地。

證據來源：design.md risks / trade-offs
Pilot（2-4 週）
單一專案試點，建立提案與任務節奏
Team（1-2 月）
擴展到跨職能小隊，建立 review gate
Org（季度）
導入 archive 與 specs 回灌機制
Slide 11 / 導入策略

建議分三階段導入：先小試，再擴大

不要一口氣全改，先用低風險方式證明有效，再全面推開。

證據來源：design.md migration plan 與 open questions

該用 OpenSpec

  • 跨角色協作頻繁
  • 需求變動風險高
  • 需要可追溯決策紀錄

可暫不使用

  • 單人短期實驗
  • 需求穩定且範圍極小
  • 尚未準備流程紀律

下一步行動

  • 挑一個中等複雜試點
  • 設定 2-3 個治理指標
  • 四週後做一次回顧

關鍵結論

  • OpenSpec 重點在可控，不在炫技
  • 流程越清楚，團隊越跑得久
  • 流程畫清楚，管理就好溝通
Slide 12 / 結語

事情越複雜，OpenSpec 的價值越明顯

如果你要的是好溝通、可追蹤、可交接，現在就可以啟動試點。
